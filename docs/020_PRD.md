# Chapter 1. Executive Summary

## 1.1 제품 정의

Project Mnemosyne은 영어 phrasal verb 학습을 첫 번째 적용 영역으로 삼는 AI 기반 Personal Learning OS이다. 이 제품은 교재를 디지털화하거나 퀴즈를 제공하는 앱이 아니라, 학습자의 기억 상태를 관찰하고, 모델링하고, 장기 기억 강화를 위한 최적의 학습 활동을 추천하는 개인 학습 운영체제이다.

제품의 중심은 콘텐츠가 아니라 Learner Model이다. 콘텐츠는 Static Dataset으로 존재하며, 학습자의 상태는 Dynamic Dataset으로 지속 업데이트된다. 같은 phrasal verb라도 학습자가 이미 안정적으로 회상할 수 있는 표현인지, 유사 표현과 혼동하는 표현인지, 뜻은 알지만 대화에서 사용하지 못하는 표현인지에 따라 학습 활동이 달라진다.

## 1.2 핵심 문제

성인 영어 학습의 핵심 문제는 콘텐츠 부족이 아니라 기억 유지 실패이다. 사용자는 좋은 교재와 앱을 사용할 수 있지만, 시간이 지나면 많은 표현을 잊고 실제 대화에서 활용하지 못한다. 특히 phrasal verb는 단순 단어보다 학습 난이도가 높다. 의미, 문법 패턴, 목적어 분리 가능성, 상황별 자연스러움, 유사 표현과의 차이가 동시에 작동하기 때문이다.

따라서 이 제품은 “더 많은 콘텐츠를 제공하는 것”이 아니라 “이미 배운 지식을 오래 기억하고 실제로 사용하게 만드는 것”을 목표로 한다.

## 1.3 핵심 솔루션

Project Mnemosyne은 콘텐츠를 Learning Object 단위로 분해하고, 각 Learning Object를 학습자의 Memory Object와 연결한다. Static Dataset은 OCR 원문, 출처, 의미, 문법, 예문, 유사 표현, 혼동 표현, 퀴즈 템플릿을 담고, Dynamic Dataset은 학습자의 정답률, 오답 패턴, 응답 시간, 복습 이력, Memory Strength, Mastery Score, Forgetting Risk를 담는다.

Adaptive Engine은 이 데이터를 바탕으로 Daily Session을 구성한다. 사용자는 “오늘 어떤 Unit을 공부할지” 선택하지 않는다. 시스템이 “오늘 가장 기억 강화 효과가 높은 학습 활동”을 제안한다.

## 1.4 MVP 목표

MVP의 목표는 완전한 상용 앱이 아니라, 개인 학습 루프가 실제로 작동하는 최소 시스템을 만드는 것이다. MVP는 다음 기능을 포함해야 한다. 첫째, OCR 기반 Dataset을 Learning Object 단위로 재구성한다. 둘째, Review Queue와 Mastery Score를 구현한다. 셋째, Daily Session을 신규 학습보다 복습 우선으로 구성한다. 넷째, Recognition보다 Retrieval 중심의 문제 유형을 제공한다. 다섯째, 학습 이벤트를 기록하고 Learner Model을 업데이트한다. 여섯째, AI Tutor가 정답 직접 제공보다 힌트, 비교, 회상 유도 중심으로 동작하게 한다.

## 1.5 제외 범위

MVP에서는 상용 결제, 다중 사용자 계정, 커뮤니티, 리더보드, 캐릭터형 게임화, 앱스토어 배포, 고급 음성 평가, 네이티브 모바일 앱 개발을 제외한다. 이 프로젝트는 상용화를 목표로 하지 않으며, 정도님 한 명의 학습 효율과 지속성을 극대화하는 개인 시스템으로 설계한다.

## 1.6 North Star Metric

제품의 North Star는 Long-term Usable Knowledge이다. 이는 단순히 학습한 표현의 수가 아니라, 일정 시간이 지난 뒤에도 사용자가 직접 회상하고 실제 문장이나 대화에서 사용할 수 있는 표현의 수를 의미한다. 완료 Unit 수, 당일 정답률, 학습 시간은 보조 지표이며 최종 성공 기준이 아니다.

# Chapter 2. Product Context

## 2.1 기존 학습 앱의 한계

기존 영어 학습 앱은 크게 네 가지 유형으로 나뉜다. 첫째, 교재형 앱은 책의 내용을 디지털 화면으로 옮긴다. 둘째, 퀴즈형 앱은 문제 풀이와 정답률을 중심으로 학습을 구성한다. 셋째, 플래시카드형 앱은 반복 노출과 간격 반복을 제공한다. 넷째, AI 회화형 앱은 대화 경험을 제공한다. 각각 장점이 있지만, 대부분 학습자의 기억 상태를 제품의 중심 데이터로 삼지는 않는다.

특히 많은 앱은 사용자가 “학습했다”는 사실과 “기억하고 있다”는 사실을 구분하지 않는다. 또한 “문제를 맞혔다”는 사실과 “실제 상황에서 사용할 수 있다”는 사실도 구분하지 않는다. Project Mnemosyne은 이 간극을 핵심 문제로 본다.

## 2.2 성인 학습자의 제약

성인 학습자는 학생과 다르다. 학습 시간은 제한적이고, 업무 피로도와 일정 변동이 크며, 학습 목표가 시험 점수에만 있지 않다. 특히 정도님의 경우 영어 학습은 업무 커뮤니케이션, 해외 전문가와의 대화, 전략적 토론, 산업 리서치와 연결될 수 있다. 따라서 제품은 장시간 학습 세션이나 강제 진도보다 짧고 효율적인 학습 세션, 즉시 회상 가능한 표현, 실제 업무 맥락에 적용 가능한 예문을 우선해야 한다.

성인 학습자는 과도한 게임화에 피로를 느낄 수 있다. 배지, 포인트, 캐릭터, 리그 시스템은 일부 사용자에게 효과적일 수 있으나, 이 프로젝트의 1차 사용자는 실질적인 학습 성과와 시간 효율을 더 중요하게 본다. 따라서 UX는 장식보다 명확성, 몰입보다 효율, 재미보다 지속 가능성을 우선한다.

## 2.3 Phrasal Verb 학습의 특수성

Phrasal verb는 일반 단어 암기와 다르다. 예를 들어 `look up`, `look into`, `look over`, `look for`는 모두 `look`을 포함하지만 의미와 사용 맥락이 다르다. 학습자는 단일 표현의 뜻을 아는 것만으로 충분하지 않고, 유사 표현을 구분하고, 문장 구조 안에서 올바르게 사용하고, 상황에 맞는 자연스러운 표현을 선택할 수 있어야 한다.

또한 일부 phrasal verb는 separable 여부, 목적어 위치, formal/informal 사용감, 구어체 빈도, 비즈니스 맥락 적합성 등의 속성을 가진다. 따라서 Dataset은 단순히 `verb`, `meaning`, `example`만 포함해서는 안 되고, 문법 패턴, 의미군, 상황 태그, 혼동 표현, 예문 유형까지 구조화해야 한다.

## 2.4 왜 AI가 필요한가

AI는 단순 문제 생성기가 아니다. 이 프로젝트에서 AI가 필요한 이유는 세 가지이다. 첫째, AI는 Learning Object를 다양한 상황과 문장으로 재표현하여 학습자의 적용 능력을 강화할 수 있다. 둘째, AI는 학습자의 오답과 응답 패턴을 바탕으로 혼동 원인을 설명하고 다음 활동을 추천할 수 있다. 셋째, AI는 정답을 바로 알려주는 대신 힌트, 비교, 질문을 통해 회상을 유도할 수 있다.

다만 AI는 사용자의 모든 개인 정보를 알 수 없으며, 알아서도 안 된다. 이 제품의 AI는 앱 내부 학습 데이터만을 근거로 개입한다. 이 제약은 제품의 안전성과 설명 가능성을 높인다.

# Chapter 3. User Definition

## 3.1 기본 사용자 정의

Project Mnemosyne의 1차 사용자는 정도님 한 명이다. 본 제품은 상용화를 목표로 하지 않으며, 불특정 다수의 평균적 학습자를 만족시키는 방향으로 설계하지 않는다. 제품은 정도님의 학습 목표, 사용 습관, 업무 맥락, 선호하는 학습 방식에 맞춰 최적화된다.

사용자는 전략컨설팅 및 반도체 산업 전략 분야에서 일하며, 영어 학습의 목적은 시험이 아니라 실사용이다. 주요 사용 상황은 해외 전문가 미팅, 산업 리서치, 비즈니스 대화, 기술 및 전략 논의, 영어 자료 이해와 표현력 개선이다. 따라서 학습 콘텐츠는 일반 영어 표현에서 시작하되, 장기적으로는 비즈니스 및 산업 맥락으로 확장 가능해야 한다.

## 3.2 사용자 니즈

사용자는 학습할 콘텐츠를 스스로 매번 고르고 싶지 않다. 사용자는 제한된 시간 안에서 오늘 가장 필요한 학습만 수행하고 싶다. 사용자는 단순 정답률보다 실제로 기억하고 사용할 수 있는지를 중요하게 본다. 사용자는 앱이 학습 진행 상황을 기록하고, 약한 표현과 잊어가는 표현을 자동으로 찾아주기를 기대한다.

또한 사용자는 성인 학습자이므로 학습 제품이 과도하게 유치하거나 게임화되는 것을 원하지 않는다. 앱은 명확하고 차분하며 효율적이어야 한다. 화면은 복잡하지 않아야 하고, 매일 해야 할 학습 행동은 분명해야 한다.

## 3.3 사용자 제약

사용자는 매일 긴 시간을 투자하기 어렵다. 따라서 Daily Session은 10~15분을 기본값으로 설계한다. 사용자가 더 길게 학습할 수 있는 날에는 추가 세션을 제공할 수 있지만, 기본 설계는 짧은 고효율 세션이어야 한다. 또한 사용자의 외부 일정이나 피로도는 시스템이 알 수 없으므로, 학습 강도 조절은 앱 내부 행동 데이터에 근거해야 한다.

## 3.4 사용자 성공 상태

사용자 성공 상태는 “교재 완료”가 아니다. 성공 상태는 사용자가 일정 시간이 지난 후에도 주요 phrasal verb를 직접 회상하고, 문장에 적용하고, 실제 대화에서 사용할 수 있는 상태이다. 예를 들어 사용자가 `put off`, `bring up`, `go through`, `come across`, `look into` 같은 표현을 상황에 맞게 자연스럽게 사용할 수 있다면 이는 제품 성공에 가까운 상태이다.

## 3.5 사용자 경험 문장

이 제품이 잘 작동할 때 사용자는 다음과 같이 느껴야 한다. “오늘 무엇을 해야 할지 고민하지 않아도 된다.” “내가 잊어가는 표현을 앱이 알고 있다.” “문제를 많이 푼다는 느낌보다 실제로 기억이 강화된다는 느낌이 든다.” “예전에 배운 표현이 실제 대화에서 떠오른다.” “앱이 나를 가르치는 것이 아니라 내 기억을 관리해준다.”

# Chapter 4. Problem Statement

## 4.1 문제 1: 학습과 기억의 분리

대부분의 학습 시스템은 학습 활동을 기록하지만 기억 상태를 충분히 모델링하지 않는다. 사용자가 어떤 Unit을 완료했는지, 몇 문제를 맞혔는지는 쉽게 기록할 수 있다. 그러나 사용자가 30일 뒤 그 표현을 회상할 수 있는지, 유사 표현과 혼동하지 않는지, 실제 대화에서 사용할 수 있는지는 별도의 모델링이 필요하다.

Project Mnemosyne은 이 문제를 해결하기 위해 학습 완료 상태와 기억 상태를 분리한다. Learning Object는 콘텐츠 단위이며, Memory Object는 사용자의 기억 상태 단위이다. 제품은 Learning Object를 보여주는 데서 끝나지 않고, 각 Memory Object의 상태를 지속적으로 업데이트해야 한다.

## 4.2 문제 2: Recognition 중심 학습

객관식 문제는 학습자가 정답을 알아보는 능력을 측정한다. 그러나 실제 사용 상황에서는 보기 중에서 정답을 고르는 것이 아니라, 아무 단서 없이 표현을 떠올려야 한다. 따라서 Recognition 중심 학습은 초기 노출에는 유용하지만 장기 목표가 될 수 없다.

제품은 Retrieval Practice를 중심으로 문제 유형을 설계해야 한다. 뜻을 보고 표현을 입력하기, 상황을 보고 표현을 말하기, 문장을 직접 완성하기, 유사 표현 중 왜 특정 표현이 더 자연스러운지 설명하기 같은 활동이 핵심이어야 한다.

## 4.3 문제 3: 고정 커리큘럼의 한계

책의 Unit 순서는 저자와 출판 편집자가 설계한 구조이다. 이는 학습 콘텐츠를 체계적으로 제공하는 데는 유용하지만, 개별 학습자의 기억 상태를 반영하지 않는다. 사용자가 Unit 6을 공부해야 하는 이유는 Unit 5를 끝냈기 때문이 아니라, 현재 사용자의 기억 상태상 Unit 6에 포함된 Learning Object가 적절하기 때문이어야 한다.

따라서 제품은 Unit을 출처와 검수 목적으로만 유지하고, 실제 학습 경로는 Adaptive Engine이 결정해야 한다.

## 4.4 문제 4: 실제 사용으로 연결되지 않는 학습

Phrasal verb는 의미를 아는 것만으로 충분하지 않다. 학습자는 표현을 문장 안에서 쓰고, 유사 표현과 구분하고, 대화 상황에서 사용할 수 있어야 한다. 기존 문제 풀이 중심 앱은 이 전환을 충분히 지원하지 못한다. Project Mnemosyne은 학습 단계를 Exposure, Recognition, Retrieval, Application, Conversation, Automaticity로 정의하고, 각 Learning Object가 어느 단계에 있는지를 상태로 관리해야 한다.

## 4.5 문제 5: AI의 잘못된 사용

AI는 학습 앱에서 쉽게 “설명 생성기”나 “문제 생성기”로 사용될 수 있다. 그러나 이 방식은 제품의 핵심 차별화를 만들지 못한다. AI는 학습자의 기억 상태를 읽고, 혼동 패턴을 분석하고, 회상을 유도하고, 다음 학습 행동을 추천하는 역할을 해야 한다. AI가 정답을 너무 빨리 알려주면 학습자의 Retrieval 기회를 빼앗을 수 있다. 따라서 AI Tutor는 설명보다 질문과 힌트를 우선해야 한다.

# Chapter 5. Product Vision

## 5.1 Personal Learning OS

Project Mnemosyne은 Personal Learning OS이다. OS라는 표현을 쓰는 이유는 이 제품이 단일 기능 앱이 아니라 학습 활동을 운영하는 기반 시스템이기 때문이다. OS는 콘텐츠를 저장하고, 학습 상태를 추적하고, 다음 행동을 스케줄링하고, AI Tutor를 호출하고, 학습 이벤트를 기록하고, 장기 기억을 최적화한다.

## 5.2 Memory-centered Product

제품의 중심 화면은 콘텐츠 목록이 아니라 기억 상태여야 한다. 사용자는 “Unit 6 보기”보다 “오늘 복습해야 할 표현”, “최근 잊어가는 표현”, “대화 적용이 약한 표현”, “혼동 중인 표현 쌍”을 중심으로 학습해야 한다. 이 구조는 사용자가 콘텐츠를 소비하는 방식에서 기억을 관리하는 방식으로 경험을 전환한다.

## 5.3 AI Coach

AI Tutor는 코치로 동작한다. 코치는 학습자 대신 답을 말하지 않는다. 코치는 힌트를 주고, 비교를 유도하고, 실수를 분석하고, 다시 시도하게 한다. 사용자가 틀렸을 때 AI는 단순히 정답을 알려주는 것이 아니라, “왜 이 표현이 아닌지”, “어떤 표현과 헷갈렸는지”, “이 상황에서 더 자연스러운 표현은 무엇인지”를 설명해야 한다.

## 5.4 Adaptive Daily Session

Daily Session은 제품의 핵심 경험이다. 사용자가 앱을 열면 시스템은 오늘의 학습 세션을 자동 구성한다. 기본 구조는 Review First, Weak Object Focus, Limited New Learning, Retrieval Practice, Short Reflection이다. 신규 학습은 복습 대기열이 안정적일 때만 제한적으로 제공한다.

## 5.5 Long-term Vision

장기적으로 Project Mnemosyne은 phrasal verb를 넘어 비즈니스 영어, 회의 영어, 협상 표현, 기술 토론 표현, 산업 리서치 표현까지 확장할 수 있다. 그러나 모든 확장은 Learning Object와 Memory Object 구조를 유지해야 한다. 콘텐츠 영역이 바뀌어도 핵심 엔진은 동일해야 한다.

# Chapter 6. Learning Philosophy

## 6.1 학습 철학의 기본 전제

Project Mnemosyne의 학습 철학은 “많이 노출되면 배운다”가 아니라 “적절한 시점에 다시 꺼내야 기억된다”는 전제에서 출발한다. 학습자는 표현을 한 번 읽는다고 기억하지 않으며, 문제를 한 번 맞힌다고 사용할 수 있는 것도 아니다. 기억은 반복 노출보다 Retrieval, Contextual Application, Error Correction, Spaced Review를 통해 강화된다.

따라서 제품은 학습자의 활동량을 늘리는 방식으로 설계하지 않는다. 제품은 제한된 시간 안에서 가장 기억 강화 효과가 높은 활동을 선택해야 한다. 이 원칙은 Daily Session, Review Queue, Quiz Type, AI Tutor Interaction, Dataset 구조 전체에 적용된다.

## 6.2 학습 단계 모델

모든 Learning Object는 다음 6단계 학습 모델 중 하나 이상의 상태를 가진다.

```text
Exposure → Recognition → Retrieval → Application → Conversation → Automaticity
```

Exposure는 표현을 처음 접하는 단계이다. Recognition은 뜻이나 형태를 보고 알아보는 단계이다. Retrieval은 단서가 제한된 상태에서 직접 떠올리는 단계이다. Application은 문장 안에서 적절하게 적용하는 단계이다. Conversation은 실제 대화 맥락에서 자연스럽게 사용하는 단계이다. Automaticity는 의식적 노력 없이 표현이 떠오르는 단계이다.

제품은 사용자가 특정 표현을 어느 단계까지 도달했는지 추정해야 한다. 예를 들어 객관식 의미 문제를 맞혔다고 해서 Retrieval 단계에 도달했다고 판단해서는 안 된다. 의미를 보고 표현을 직접 입력하거나, 상황 설명을 보고 적절한 표현을 생성할 수 있어야 Retrieval 단계로 인정할 수 있다.

## 6.3 Retrieval Practice 중심 설계

Retrieval Practice는 Project Mnemosyne의 핵심 학습 원칙이다. 사용자가 정답을 알아보는 것보다 직접 떠올리는 것이 장기 기억 형성에 더 중요하다. 따라서 제품의 문제 유형은 초기에는 Recognition을 허용하되, 반복 학습이 진행될수록 Retrieval 비중을 높여야 한다.

문제 유형은 다음 순서로 난이도가 상승해야 한다.

| 단계 | 문제 유형 | 목적 |
|---|---|---|
| 1 | 의미 인식 | 표현과 의미의 기본 연결 형성 |
| 2 | 예문 선택 | 문맥 속 의미 확인 |
| 3 | 빈칸 완성 | 제한된 단서 기반 회상 |
| 4 | 직접 입력 | 능동 회상 |
| 5 | 상황 적용 | 의미와 맥락 연결 |
| 6 | 대화 생성 | 실제 사용 능력 강화 |

## 6.4 Spaced Review 원칙

복습은 고정 간격이 아니라 Memory Strength와 Forgetting Risk에 의해 결정되어야 한다. 단순히 1일, 3일, 7일, 14일 같은 고정 간격만 사용하는 것은 초기 MVP에서는 가능하지만, 최종 구조에서는 학습자의 실제 정답률, 응답 시간, 오답 패턴, 복습 지연을 반영해야 한다.

예를 들어 어떤 표현을 사용자가 빠르게 맞히고 대화에서도 자연스럽게 사용했다면 복습 간격을 늘릴 수 있다. 반대로 객관식은 맞히지만 직접 입력에서 실패하거나 유사 표현과 반복적으로 혼동한다면 복습 간격을 줄여야 한다. 복습 간격은 “맞혔는가”만이 아니라 “얼마나 안정적으로 회상했는가”에 의해 조정되어야 한다.

## 6.5 Interleaving 원칙

Phrasal verb 학습에서는 유사 표현 간 혼동이 핵심 문제이다. 따라서 같은 의미군 또는 같은 동사 기반 표현을 분리해서 배우기보다, 적절한 시점에 서로 비교해야 한다. 예를 들어 `look up`, `look into`, `look over`, `look for`는 각각 별도로 암기하는 것보다 비교 학습을 통해 차이를 명확히 해야 한다.

Interleaving은 무작위 혼합이 아니다. 학습자가 특정 표현을 어느 정도 인식한 뒤, 혼동 가능성이 높은 표현과 함께 제시해야 한다. 너무 이른 비교는 Cognitive Load를 높이고, 너무 늦은 비교는 잘못된 기억을 고착시킬 수 있다. Adaptive Engine은 혼동 표현을 적절한 시점에 묶어 제공해야 한다.

## 6.6 Error-based Learning

오답은 실패가 아니라 학습 데이터이다. 사용자가 어떤 표현을 틀렸는지보다 왜 틀렸는지가 중요하다. 예를 들어 `put off`를 `put away`로 잘못 선택했다면 이는 단순 오답이 아니라 semantic confusion이다. `look up it`처럼 어순을 틀렸다면 grammar pattern error이다. 뜻은 맞지만 문장 맥락에 부자연스럽게 사용했다면 pragmatic error이다.

제품은 오답을 유형화해야 한다. 오답 유형은 Learner Model과 Adaptive Engine에 반영되어 다음 학습 활동을 결정해야 한다. 모든 오답은 최소한 다음 중 하나로 분류되어야 한다: meaning error, grammar error, collocation error, context error, similar expression confusion, spelling error, production failure.

## 6.7 Desirable Difficulty

학습은 너무 쉬우면 기억을 강화하지 못하고, 너무 어려우면 지속성을 해친다. Project Mnemosyne은 사용자가 약간 어렵지만 성공 가능한 수준의 문제를 풀도록 해야 한다. 이를 Desirable Difficulty로 정의한다.

문제 난이도는 표현 자체의 난이도뿐 아니라 학습자의 현재 상태에 따라 달라진다. 이미 잘 아는 표현에 대해 객관식 문제를 반복하는 것은 지나치게 쉽다. 반대로 처음 보는 표현에 대해 대화 생성 문제를 요구하는 것은 지나치게 어렵다. Adaptive Engine은 Learning Stage, Memory Strength, Recent Accuracy, Response Latency를 기반으로 적절한 난이도를 선택해야 한다.

# Chapter 7. Product Scope

## 7.1 전체 범위

Project Mnemosyne의 전체 범위는 Static Dataset, Dynamic Learner Model, Adaptive Engine, Review Engine, AI Tutor, Minimal Learning Interface를 포함한다. 이 여섯 요소가 함께 작동할 때 제품의 핵심 가치가 발생한다. 단순히 Dataset만 만들거나 퀴즈 UI만 만드는 것은 제품의 일부일 뿐이며, 최종 목표를 달성하기에 충분하지 않다.

## 7.2 MVP 범위

MVP는 제품 철학을 검증하기 위한 최소 구현이다. MVP는 다음을 포함한다.

| 영역 | MVP 요구사항 |
|---|---|
| Dataset | OCR 기반 데이터를 Learning Object로 재구성 |
| Source Reference | Unit, Page, Exercise를 Provenance로 유지 |
| Quiz | Recognition, Retrieval 중심 문제 제공 |
| Review | Review Queue 생성 및 복습 우선 처리 |
| Learner Model | Mastery Score, Memory Strength, Last Reviewed 관리 |
| Event Log | 모든 학습 활동 기록 |
| AI Tutor | 힌트, 비교, 회상 유도 중심의 기본 코칭 |
| UI | Today Mission 중심의 단순 인터페이스 |

## 7.3 MVP 제외 범위

MVP에서는 상용화 관련 기능을 제외한다. 결제, 사용자 가입, 다중 사용자 관리, 커뮤니티, 리더보드, 앱스토어 배포, 마케팅 페이지는 필요하지 않다. 또한 고급 음성 인식과 발음 평가는 중요하지만 초기 MVP에서는 제외 가능하다. Speaking 기능은 텍스트 기반 대화 생성과 문장 작성으로 우선 대체할 수 있다.

## 7.4 V1 범위

V1은 MVP의 학습 루프가 작동한다는 전제하에 Adaptive Engine을 강화하는 단계이다. V1에서는 혼동 표현 기반 Interleaving, 오답 유형 분류, 학습 단계별 문제 전환, Review Interval 자동 조정, AI Tutor의 개인화 피드백을 강화한다. 또한 Dataset의 Knowledge Graph를 확장하여 의미군, 문법 패턴, 비즈니스 맥락 예문을 추가한다.

## 7.5 V2 범위

V2는 Conversation 중심 확장 단계이다. 사용자는 단순 문제 풀이를 넘어, 오늘 복습한 표현을 짧은 대화 안에서 사용해야 한다. AI Tutor는 대화 상대 역할을 하며, 사용자가 표현을 자연스럽게 사용할 수 있도록 상황을 제공한다. 이 단계에서는 Conversation Success Rate와 Expression Usage Score를 추가로 추적한다.

## 7.6 V3 범위

V3는 Personal Learning OS로 확장하는 단계이다. Phrasal verb 외에도 idiom, business expression, meeting English, negotiation English, technical discussion expression을 같은 Learning Object 구조로 추가한다. 이때 중요한 것은 콘텐츠 확장이 아니라 엔진 재사용성이다. 새로운 콘텐츠가 들어와도 Memory Engine, Adaptive Engine, AI Tutor 구조는 유지되어야 한다.

# Chapter 8. Core User Journey

## 8.1 기본 사용 시나리오

사용자는 앱을 열고 오늘의 학습 세션을 시작한다. 앱은 사용자의 전체 학습 데이터와 Review Queue를 기반으로 오늘의 Mission을 제안한다. 사용자는 Unit 목록을 탐색하지 않고, 시스템이 제시한 우선순위에 따라 학습한다. 세션이 끝나면 앱은 오늘 강화된 표현, 아직 약한 표현, 다음 복습 예정 표현을 간단히 보여준다.

기본 흐름은 다음과 같다.

```text
앱 실행 → Today Mission 확인 → Review First → Weak Object Practice → Limited New Learning → Short Reflection → Next Review Queue 업데이트
```

## 8.2 Today Mission

Today Mission은 첫 화면의 중심이다. Today Mission은 단순히 “오늘 10개 학습”이 아니라, 왜 오늘 이 학습이 필요한지를 설명해야 한다. 예를 들어 “최근 `put off`와 `put away`를 혼동하고 있어 비교 연습을 우선합니다”, “어제 신규 학습한 표현 4개가 초기 망각 구간에 있어 복습이 필요합니다”, “오늘은 복습 대기열이 많아 신규 학습을 제한합니다”와 같은 설명이 필요하다.

## 8.3 Review First Flow

Review First는 제품의 핵심 흐름이다. 사용자가 앱을 열면 신규 학습보다 복습을 먼저 수행한다. Review Queue는 Forgetting Risk, Memory Strength, Last Reviewed, Error Severity, Confusion Level을 기준으로 정렬된다. 사용자는 가장 중요한 복습 항목부터 학습한다.

## 8.4 New Learning Flow

신규 학습은 복습 부담이 안정적인 경우에만 제공한다. 신규 Learning Object는 단순히 다음 Unit에서 선택하지 않고, 기존 Knowledge Graph와 연결성이 높은 표현, 현재 학습자의 약점과 관련된 표현, 난이도상 적절한 표현을 기준으로 선택한다. 신규 학습은 짧아야 하며, 한 번에 너무 많은 표현을 도입하지 않는다.

## 8.5 Weak Object Flow

Weak Object는 학습자가 반복적으로 틀리거나, 응답 시간이 길거나, 유사 표현과 혼동하거나, 대화 적용에 실패하는 표현이다. Weak Object Flow에서는 단순 반복보다 원인 기반 학습을 제공해야 한다. 예를 들어 meaning confusion이면 유사 표현 비교를, grammar error이면 문장 구조 연습을, production failure이면 직접 입력과 대화 적용을 제공한다.

## 8.6 Session End Reflection

세션 종료 화면은 점수판이 아니라 기억 상태 요약이어야 한다. 사용자는 오늘 몇 문제를 맞혔는지보다 어떤 표현의 Memory Strength가 올라갔는지, 어떤 표현이 여전히 약한지, 다음 복습이 언제인지 알아야 한다. 세션 종료 후 앱은 다음 Review Queue를 업데이트한다.

# Chapter 9. Learning Object Architecture

## 9.1 Learning Object 정의

Learning Object는 제품이 학습 대상으로 관리하는 최소 지식 단위이다. Phrasal verb 하나가 기본 Learning Object가 될 수 있지만, 하나의 phrasal verb가 여러 의미를 가진 경우 의미별로 별도 Learning Object를 생성할 수 있다. 예를 들어 `take off`가 “이륙하다”, “옷을 벗다”, “성공하기 시작하다”의 의미를 가진다면 각 의미는 별도 Learning Object로 분리하는 것이 바람직하다.

## 9.2 Learning Object 필수 속성

각 Learning Object는 다음 속성을 가져야 한다.

| 필드 | 설명 |
|---|---|
| learning_object_id | 고유 ID |
| expression | 표현 |
| base_verb | 기본 동사 |
| particle | particle 또는 particles |
| meaning | 핵심 의미 |
| korean_explanation | 한국어 설명 |
| grammar_pattern | 문법 패턴 |
| separability | 목적어 분리 가능 여부 |
| transitivity | 타동사/자동사 여부 |
| examples | 예문 |
| source_reference | Unit, Page, Exercise 출처 |
| semantic_tags | 의미 태그 |
| context_tags | 상황 태그 |
| confusing_objects | 혼동 가능 표현 |
| quiz_templates | 문제 템플릿 |
| ai_prompt_refs | AI Tutor 프롬프트 참조 |

## 9.3 Source Reference

Unit, Page, Exercise는 Learning Object의 Primary Key가 아니다. Source Reference는 출처와 검수 목적의 Provenance이다. 이는 OCR 오류 검수, 저작권 범위 관리, 정답지 연결, 원문 추적을 위해 필요하다. 그러나 사용자에게 학습 순서를 강제하는 기준으로 사용해서는 안 된다.

## 9.4 Knowledge Graph

Knowledge Graph는 Learning Object 간 관계를 표현한다. 관계 유형은 다음을 포함한다.

| 관계 유형 | 예시 |
|---|---|
| same_base_verb | look up ↔ look into |
| similar_meaning | find out ↔ figure out |
| opposite_meaning | put on ↔ take off |
| often_confused | put off ↔ put away |
| same_context | meeting, research, travel |
| grammar_related | separable transitive verbs |
| business_relevant | bring up, follow up, lay out |

Knowledge Graph는 Adaptive Engine의 핵심 입력값이다. 학습자가 특정 표현을 틀렸을 때, 시스템은 연결된 혼동 표현과 문법 패턴을 함께 고려해야 한다.

## 9.5 Memory Object와의 관계

Learning Object는 정적 지식 단위이고, Memory Object는 학습자별 동적 기억 상태이다. 같은 Learning Object라도 사용자의 Memory Strength, Mastery Score, Error History, Review Interval은 계속 변한다. 따라서 Learning Object와 Memory Object는 분리하되, 항상 연결되어야 한다.

# Chapter 10. Learner Model

## 10.1 Learner Model 정의

Learner Model은 사용자의 현재 학습 상태를 표현하는 동적 모델이다. 이 모델은 사용자의 외부 개인 정보를 포함하지 않고, 앱 내부 학습 행동 데이터만을 기반으로 한다. Learner Model은 Adaptive Engine이 다음 학습 활동을 선택하는 핵심 입력값이다.

## 10.2 핵심 상태값

Learner Model은 최소한 다음 상태값을 관리해야 한다.

| 상태값 | 설명 |
|---|---|
| mastery_score | 해당 표현에 대한 종합 숙련도 |
| memory_strength | 현재 기억 강도 |
| forgetting_risk | 가까운 시점에 잊을 가능성 |
| last_reviewed_at | 마지막 복습 시점 |
| next_review_at | 다음 복습 예정 시점 |
| correct_rate | 누적 및 최근 정답률 |
| response_latency | 응답 시간 |
| error_history | 오답 이력 |
| confusion_pairs | 반복 혼동 표현 |
| learning_stage | Exposure~Automaticity 단계 |
| conversation_success | 대화 적용 성공 수준 |

## 10.3 Mastery Score

Mastery Score는 단순 정답률이 아니다. Mastery Score는 Recognition, Retrieval, Application, Conversation 단계의 성과를 종합해 계산해야 한다. 예를 들어 객관식 문제를 잘 맞혀도 직접 입력과 대화 적용에 실패하면 Mastery Score는 높게 평가해서는 안 된다. 반대로 응답 시간이 빠르고 다양한 문맥에서 자연스럽게 사용할 수 있다면 Mastery Score는 높아져야 한다.

## 10.4 Memory Strength

Memory Strength는 현재 해당 표현이 얼마나 안정적으로 기억되어 있는지를 나타낸다. Memory Strength는 최근 정답 여부, 응답 시간, 문제 난이도, 복습 간격, 오답 심각도에 의해 업데이트된다. Memory Strength가 낮거나 빠르게 하락하는 표현은 Review Queue에서 우선순위가 높아진다.

## 10.5 Forgetting Risk

Forgetting Risk는 가까운 시점에 잊을 가능성이다. Last Reviewed 이후 경과 시간, 과거 Forgetting Pattern, 최근 성공/실패 이력, 학습 단계, 표현 난이도를 기반으로 추정한다. 초기 MVP에서는 단순 규칙 기반으로 구현할 수 있으나, 향후 Event Log가 충분히 쌓이면 더 정교한 모델로 확장할 수 있다.

## 10.6 Learner Model 업데이트

모든 학습 이벤트는 Learner Model을 업데이트해야 한다. 이벤트에는 문제 응답, 정답 여부, 응답 시간, 힌트 사용, 재시도 여부, AI Tutor 피드백, 대화 적용 결과, 사용자가 스킵한 활동이 포함된다. 특히 스킵도 중요한 데이터이다. 반복적으로 Speaking 또는 직접 입력 문제를 스킵한다면 해당 활동의 난이도나 부담이 높을 수 있다.

## 10.7 Privacy Boundary

Learner Model은 학습 앱 내부 데이터만 사용한다. 시스템은 사용자의 외부 일정, 건강 상태, 업무 피로도, 위치, 메시지, 이메일 등과 연결되지 않는다. 추천은 학습 성과 기반으로만 이루어진다. 이 원칙은 설계 단순성, 프라이버시, 설명 가능성을 모두 높인다.

# Chapter 11. Adaptive Engine

## 11.1 Adaptive Engine 정의

Adaptive Engine은 Project Mnemosyne의 핵심 의사결정 시스템이다. 이 엔진은 사용자가 다음에 무엇을 학습해야 하는지, 어떤 방식으로 복습해야 하는지, 신규 학습을 허용할지, 특정 표현을 대화 연습으로 전환할지, 혹은 잠시 제외할지를 결정한다. Adaptive Engine은 단순 추천 기능이 아니라 제품의 학습 운영 로직이다.

엔진의 기본 입력값은 Static Dataset의 Learning Object와 Dynamic Dataset의 Learner Model이다. Learning Object는 표현의 의미, 문법, 예문, 난이도, 유사 표현, 출처를 제공하고, Learner Model은 사용자의 기억 강도, 정답률, 응답 시간, 오답 이력, 혼동 패턴, 복습 지연을 제공한다. Adaptive Engine은 이 두 데이터를 결합하여 학습 우선순위를 계산한다.

## 11.2 Adaptive Engine의 기본 원칙

Adaptive Engine은 네 가지 원칙을 따른다. 첫째, 복습은 신규 학습보다 우선한다. 둘째, 기억에서 사라질 위험이 높은 항목은 우선순위가 높다. 셋째, 단순 정답률보다 학습 단계와 문제 유형을 함께 고려한다. 넷째, 추천은 설명 가능해야 한다. 사용자가 왜 특정 표현을 오늘 복습해야 하는지 이해할 수 없다면, 엔진의 추천 품질은 충분하지 않다.

## 11.3 입력 데이터

Adaptive Engine은 최소한 다음 데이터를 입력으로 사용한다.

| 입력 데이터 | 설명 |
|---|---|
| Learning Object Metadata | 표현, 의미, 문법, 난이도, 유사 표현, 출처 |
| Memory Strength | 현재 기억 강도 |
| Mastery Score | 종합 숙련도 |
| Forgetting Risk | 망각 위험도 |
| Last Reviewed | 마지막 복습 시점 |
| Response Latency | 응답 시간 |
| Error History | 오답 유형과 반복성 |
| Confusion Pairs | 혼동 표현 쌍 |
| Learning Stage | Exposure~Automaticity 단계 |
| Session History | 최근 세션 성과와 스킵 패턴 |

## 11.4 추천 우선순위

추천 우선순위는 단순 점수 하나로만 결정하지 않는다. 초기 MVP에서는 규칙 기반 점수 모델을 사용하되, 장기적으로는 가중치를 조정할 수 있는 구조로 설계한다. 기본 우선순위는 다음과 같은 요소를 반영한다.

```text
priority_score =
  forgetting_risk_weight
+ low_memory_strength_weight
+ recent_error_weight
+ confusion_weight
+ overdue_review_weight
+ learning_stage_weight
- fatigue_proxy_penalty
```

`fatigue_proxy_penalty`는 외부 피로도 정보가 아니라 앱 내부 데이터에서 추정한다. 예를 들어 최근 세션에서 응답 시간이 급격히 증가하거나, 스킵이 많거나, 쉬운 문제에서 오답이 증가하면 세션 강도를 낮춘다.

## 11.5 추천 유형

Adaptive Engine은 단순히 표현 하나를 고르지 않는다. 표현과 학습 활동을 함께 추천해야 한다.

| 추천 유형 | 예시 |
|---|---|
| Review | “이 표현은 망각 위험이 높으므로 복습” |
| Contrast | “이 두 표현을 반복 혼동하므로 비교 학습” |
| Retrieval | “의미 인식은 안정적이므로 직접 회상 문제로 전환” |
| Application | “직접 회상은 가능하므로 문장 생성으로 전환” |
| Conversation | “응용 단계가 안정적이므로 대화 연습으로 전환” |
| Hold | “충분히 안정적이므로 이번 주 복습 제외” |

## 11.6 Explainability

모든 추천에는 추천 사유가 붙어야 한다. 예를 들어 “최근 7일간 `put off` 관련 직접 입력 문제에서 2회 실패했고, `put away`와의 혼동이 반복되어 오늘 비교 학습을 추천합니다”와 같은 설명이 필요하다. 이 설명은 사용자에게 보여줄 수도 있고, 내부 로그로 저장할 수도 있다. 중요한 것은 Adaptive Engine의 결정이 추적 가능해야 한다는 점이다.

## 11.7 MVP 구현 방식

MVP에서는 복잡한 머신러닝 모델을 사용하지 않는다. 규칙 기반 엔진으로 시작한다. 규칙 기반 구조는 빠르게 구현할 수 있고, 설명 가능성이 높으며, 개인 사용 목적에 충분하다. 이후 학습 이벤트가 충분히 쌓이면 점수 가중치 조정 또는 간단한 통계 모델로 확장한다.

# Chapter 12. AI Tutor

## 12.1 AI Tutor 정의

AI Tutor는 Project Mnemosyne의 대화형 학습 코치이다. AI Tutor는 정답을 알려주는 역할보다 학습자의 회상을 유도하고, 오답 원인을 분석하고, 유사 표현을 비교하고, 실제 문장과 대화 맥락에서 표현을 사용할 수 있도록 돕는 역할을 한다.

AI Tutor는 Adaptive Engine의 결정을 실행하는 인터페이스이기도 하다. Adaptive Engine이 “오늘 `look into`와 `look over` 비교 학습이 필요하다”고 판단하면, AI Tutor는 이를 사용자가 이해할 수 있는 학습 대화로 전환한다.

## 12.2 AI Tutor의 역할

AI Tutor는 네 가지 역할을 가진다.

| 역할 | 설명 |
|---|---|
| Coach | 정답보다 힌트와 질문으로 회상을 유도 |
| Explainer | 필요한 경우 문법과 의미 차이를 설명 |
| Diagnostician | 오답 원인을 분류하고 약점을 식별 |
| Conversation Partner | 표현을 실제 대화에 적용하도록 유도 |

AI Tutor는 학습자가 틀렸을 때 즉시 긴 설명을 제공하지 않는다. 먼저 짧은 힌트를 제공하고, 사용자가 다시 시도하게 한다. 두 번째 실패 이후에는 유사 표현 비교나 문법 패턴 설명을 제공한다.

## 12.3 AI Tutor 대화 원칙

AI Tutor의 대화는 성인 학습자에게 적합해야 한다. 지나치게 유치한 칭찬, 과도한 게임화 언어, 불필요한 감정 표현을 피한다. 피드백은 짧고 정확해야 하며, 사용자가 다음 행동을 명확히 알 수 있어야 한다.

AI Tutor는 다음 순서를 기본으로 따른다.

```text
시도 유도 → 힌트 제공 → 재시도 → 원인 설명 → 짧은 적용 연습 → 다음 복습 예약
```

## 12.4 정답 제공 원칙

AI Tutor는 정답을 너무 빨리 제공해서는 안 된다. 정답 공개 전 최소 한 번의 회상 기회를 제공해야 한다. 사용자가 명시적으로 “정답 보기”를 선택하거나 두 번 이상 실패한 경우 정답을 제공할 수 있다. 정답을 제공할 때도 단순 정답만 제시하지 않고, 왜 그 표현이 맞는지와 어떤 표현과 헷갈리기 쉬운지를 함께 설명한다.

## 12.5 오답 분석

AI Tutor는 오답을 다음 유형으로 분류해야 한다.

| 오답 유형 | 설명 |
|---|---|
| Meaning Error | 의미를 잘못 이해 |
| Grammar Pattern Error | 목적어 위치, 타동사성 등 문법 패턴 오류 |
| Confusion Error | 유사 표현과 혼동 |
| Context Error | 문맥상 부자연스러운 표현 선택 |
| Production Error | 알고 있지만 직접 생성 실패 |
| Spelling Error | 철자 오류 |
| Overgeneralization | 특정 표현을 과도하게 일반화 |

오답 유형은 Event Log와 Learner Model에 반영되어야 한다. AI Tutor의 설명은 이 오답 유형에 따라 달라져야 한다.

## 12.6 Conversation Practice

Conversation Practice는 MVP 이후 강화하되, 초기에도 텍스트 기반으로 최소 구현한다. AI Tutor는 짧은 상황을 제시하고 사용자가 목표 표현을 포함한 문장을 만들도록 유도한다. 예를 들어 “회의 일정을 미루는 상황에서 `put off`를 사용해 한 문장을 만들어보세요”와 같은 방식이다. 이후 V2에서는 실제 대화 흐름 안에서 복수의 표현을 자연스럽게 사용하도록 확장한다.

## 12.7 Prompt Library

AI Tutor의 행동은 코드에 직접 하드코딩하지 않고 Prompt Library에서 관리한다. Prompt는 목적별로 분리한다. 예를 들어 hint_prompt, contrast_prompt, error_diagnosis_prompt, conversation_prompt, review_summary_prompt를 별도 파일로 관리한다. 모든 Prompt는 버전 관리 대상이며, 변경 시 Decision Log 또는 Prompt Change Log에 기록한다.

# Chapter 13. Dataset v2

## 13.1 Dataset v2의 목적

Dataset v2는 기존 OCR 데이터를 앱이 학습 엔진으로 사용할 수 있는 구조로 재편한 데이터셋이다. 기존 Dataset이 책의 Unit과 Exercise를 중심으로 구성되었다면, Dataset v2는 Learning Object, Knowledge Graph, Quiz Template, Answer Graph, Source Reference를 중심으로 구성된다.

Dataset v2의 목적은 책을 재현하는 것이 아니다. 목적은 AI와 Adaptive Engine이 학습 판단을 할 수 있도록 충분히 구조화된 지식 기반을 만드는 것이다.

## 13.2 Dataset Layer

Dataset v2는 다음 계층으로 구성한다.

```text
Source Layer
  └── OCR Raw, Page Image, Unit, Exercise, Answer Key

Knowledge Layer
  └── Learning Object, Meaning, Grammar, Example, Context Tag

Graph Layer
  └── Similarity, Confusion, Same Verb, Same Context, Grammar Relation

Activity Layer
  └── Quiz Template, Retrieval Prompt, Application Task, Conversation Task

Learner Layer
  └── Memory Object, Mastery, Review Queue, Event Log
```

Source Layer는 원천 데이터 추적을 위한 계층이다. Knowledge Layer는 학습 대상 지식이다. Graph Layer는 표현 간 관계이다. Activity Layer는 학습 활동 생성을 위한 템플릿이다. Learner Layer는 사용자별 동적 상태이다.

## 13.3 Static Dataset

Static Dataset은 사용자의 학습 상태와 무관하게 고정되는 데이터이다. 여기에는 OCR 원문, 정제된 Learning Object, 문법 정보, 예문, 유사 표현, 혼동 표현, 정답지, 출처 정보가 포함된다. Static Dataset은 앱 업데이트 또는 수동 검수에 의해 변경되며, 학습 세션 중에는 임의로 변경되지 않는다.

## 13.4 Dynamic Dataset

Dynamic Dataset은 사용자의 학습 활동에 따라 계속 바뀌는 데이터이다. 여기에는 Memory Strength, Mastery Score, Review Interval, Error History, Confusion Frequency, Conversation Success, Last Reviewed, Next Review가 포함된다. Dynamic Dataset은 Event Log를 기반으로 업데이트된다.

## 13.5 Learning Object Schema 개요

Learning Object는 다음 구조를 가진다.

```json
{
  "learning_object_id": "lo_000001",
  "expression": "look up",
  "base_verb": "look",
  "particles": ["up"],
  "meaning": "search for information",
  "korean_explanation": "정보를 찾아보다",
  "grammar_pattern": "look something up / look up something",
  "separability": "separable",
  "transitivity": "transitive",
  "examples": [],
  "semantic_tags": ["search", "information"],
  "context_tags": ["study", "internet", "dictionary"],
  "confusing_objects": ["look into", "look for"],
  "source_reference": []
}
```

이 Schema는 최종 버전이 아니라 PRD 수준의 개요이다. 세부 JSON Schema는 `060_Data_Model.md`에서 정의한다.

## 13.6 OCR 데이터 처리 원칙

OCR 데이터는 원문 재현보다 학습 객체 추출을 우선한다. OCR 결과는 반드시 Source Reference와 연결되어야 하며, 검수 상태를 가져야 한다. OCR confidence가 낮은 항목은 Learning Object로 자동 승격하지 않고 Review Needed 상태로 남긴다. 정답지는 Answer Graph로 분리하고, 문제 데이터는 answer_ref를 통해 정답을 참조한다.

## 13.7 Answer Graph

Answer Graph는 책 뒤 정답지를 문제와 연결하는 구조이다. 문제 데이터에 정답을 직접 중복 저장하지 않고, answer_ref를 통해 정답 데이터를 참조한다. 이 구조는 OCR 정답 오류 수정, 문제 재생성, AI 생성 문제와 원문 문제의 분리를 쉽게 한다.

# Chapter 14. UX/UI Requirements

## 14.1 UX 기본 방향

Project Mnemosyne의 UI는 최소화되어야 한다. 제품의 지능은 화면 복잡성이 아니라 Adaptive Engine과 Learner Model 내부에 있어야 한다. 사용자는 많은 메뉴를 탐색하지 않고, 오늘 해야 할 학습 행동을 즉시 이해해야 한다. 따라서 첫 화면은 Today Mission 중심이어야 한다.

## 14.2 주요 화면

MVP의 주요 화면은 다음으로 제한한다.

| 화면 | 목적 |
|---|---|
| Today Mission | 오늘의 학습 추천과 이유 표시 |
| Review Session | 복습 문제 진행 |
| Learning Object Detail | 표현의 의미, 문법, 예문, 상태 확인 |
| Weak Objects | 약한 표현 목록 |
| Progress Dashboard | 기억 상태와 학습 추세 확인 |
| AI Tutor Chat | 힌트, 비교, 대화 연습 |

## 14.3 Today Mission 화면

Today Mission 화면은 앱의 홈 화면이다. 이 화면은 “오늘 할 일”과 “그 이유”를 보여준다. 예를 들어 “복습 7개, 비교 연습 2개, 신규 학습 1개”처럼 구성하되, 단순 숫자보다 추천 이유를 함께 보여주는 것이 중요하다. 사용자는 학습을 시작하기 전에 오늘 세션의 목적을 이해해야 한다.

## 14.4 Progress Dashboard

Progress Dashboard는 정답률 중심이 아니라 기억 상태 중심이어야 한다. 예를 들어 전체 Mastery, Memory Strength Distribution, Forgetting Risk가 높은 표현, 최근 회복된 표현, 대화 적용이 약한 표현을 보여준다. 완료 Unit 수는 보조 정보로만 제공한다.

## 14.5 UI 톤앤매너

UI는 성인 학습자에게 적합해야 한다. 지나치게 밝고 유치한 게임형 디자인은 피한다. 문장은 짧고 명확해야 하며, 사용자가 다음 행동을 바로 이해할 수 있어야 한다. 칭찬은 과장하지 않고, 피드백은 구체적이어야 한다.

## 14.6 세션 종료 화면

세션 종료 화면은 점수판이 아니라 기억 업데이트 요약이어야 한다. 예를 들어 “오늘 강화된 표현”, “아직 불안정한 표현”, “다음 복습 예정”, “반복 혼동 표현”을 보여준다. 사용자는 세션 후 자신이 무엇을 더 잘 기억하게 되었는지 이해해야 한다.

# Chapter 15. Data & Event Logging

## 15.1 Event Logging의 목적

Event Log는 Learner Model의 원천 데이터이다. Adaptive Learning은 학습 행동 데이터 없이는 작동할 수 없다. 따라서 모든 중요한 학습 활동은 이벤트로 기록되어야 한다. 이벤트는 향후 Mastery Score, Memory Strength, Forgetting Risk, Review Queue, AI Tutor 피드백에 사용된다.

## 15.2 필수 이벤트

MVP에서 최소한 다음 이벤트를 기록한다.

| 이벤트 | 설명 |
|---|---|
| session_started | 학습 세션 시작 |
| session_completed | 학습 세션 종료 |
| item_shown | 특정 Learning Object 제시 |
| answer_submitted | 답안 제출 |
| answer_evaluated | 정답/오답 및 평가 결과 |
| hint_requested | 힌트 요청 |
| retry_attempted | 재시도 |
| item_skipped | 항목 스킵 |
| ai_feedback_shown | AI 피드백 표시 |
| review_scheduled | 다음 복습 예약 |
| mastery_updated | Mastery Score 업데이트 |

## 15.3 이벤트 데이터 구조

각 이벤트는 최소한 timestamp, event_type, learning_object_id, user_id, session_id, activity_type, result, response_latency, metadata를 포함해야 한다. 단일 사용자 프로젝트이므로 user_id는 고정값으로 시작할 수 있지만, 데이터 구조상 user_id 필드는 유지한다.

## 15.4 응답 시간

응답 시간은 중요한 학습 신호이다. 정답을 맞혔더라도 응답 시간이 길면 기억이 약할 수 있다. 반대로 빠르고 안정적으로 답하면 Memory Strength가 높다고 볼 수 있다. 따라서 모든 답변 이벤트에는 response_latency를 기록한다.

## 15.5 스킵 데이터

스킵은 실패가 아니라 중요한 신호이다. 사용자가 특정 문제 유형을 반복적으로 스킵한다면 문제 난이도가 높거나, 부담이 크거나, 해당 활동의 UX가 좋지 않을 수 있다. 특히 Speaking 또는 직접 입력 활동의 스킵은 학습 단계 전환이 너무 이르다는 신호일 수 있다.

## 15.6 데이터 보존 원칙

Event Log는 원칙적으로 삭제하지 않는다. Dynamic State는 Event Log를 기반으로 재계산 가능해야 한다. 즉, Mastery Score 계산 방식이 변경되더라도 과거 Event Log를 다시 처리하여 새로운 상태값을 만들 수 있어야 한다. 이는 향후 알고리즘 개선에 중요하다.

# Chapter 16. Technical Architecture

## 16.1 아키텍처 원칙

Project Mnemosyne의 기술 아키텍처는 개인 사용, 빠른 반복, 데이터 투명성, AI Agent 협업을 우선한다. 상용 서비스 수준의 복잡한 인프라보다, 로컬 또는 개인 서버 환경에서 안정적으로 동작하고 Codex, Claude Code, Cursor가 쉽게 이해하고 수정할 수 있는 구조가 더 중요하다. 초기 구현은 Web-first, Local-first를 기본 방향으로 한다.

시스템은 Static Dataset과 Dynamic Dataset을 명확히 분리해야 한다. Static Dataset은 Learning Object, Knowledge Graph, Quiz Template, Source Reference처럼 콘텐츠 자체를 담고, Dynamic Dataset은 Event Log, Memory Object, Mastery Score, Review Queue처럼 학습자의 상태를 담는다. 이 분리는 향후 Dataset을 재생성하거나 알고리즘을 변경해도 학습 이력을 보존하기 위해 필수적이다.

## 16.2 권장 기술 스택

MVP 권장 기술 스택은 다음과 같다.

| 영역 | 권장안 | 이유 |
|---|---|---|
| Frontend | Next.js 또는 React | 빠른 개발, 로컬 실행, UI 확장성 |
| Backend | Node.js/TypeScript 또는 Python FastAPI | AI 연동과 데이터 처리 용이 |
| Local DB | SQLite | 개인 사용, 이식성, Event Log 관리 용이 |
| Static Dataset | JSON/Markdown | Git 관리 및 AI Agent 이해 용이 |
| AI Layer | API 기반 호출 | Prompt Library 버전 관리 |
| Package Manager | pnpm 또는 uv | 재현 가능한 개발 환경 |
| Repository | GitHub | 문서, 데이터, 코드 통합 관리 |

상용화가 목적이 아니므로 초기에는 인증, 결제, 멀티테넌시, 대규모 서버 인프라를 고려하지 않는다. 단일 사용자 환경에서 학습 루프가 제대로 작동하는지 검증하는 것이 우선이다.

## 16.3 모듈 구조

구현 저장소는 다음 모듈로 나누는 것을 권장한다.

```text
src/
  app/
  components/
  features/
    daily-session/
    review/
    learning-object/
    ai-tutor/
    dashboard/
  core/
    adaptive-engine/
    memory-engine/
    scoring/
    scheduler/
  data/
    static-loader/
    event-store/
    learner-state/
  ai/
    prompts/
    tutor-client/
  types/
  utils/
dataset/
  static/
  source/
  graph/
  templates/
docs/
schemas/
```

`core/`는 제품의 학습 지능이 들어가는 영역이며 UI와 분리해야 한다. UI 변경이 Memory Engine이나 Adaptive Engine을 오염시켜서는 안 된다.

## 16.4 데이터 저장 방식

초기 구현은 SQLite와 JSON 파일을 혼합한다. Static Dataset은 JSON 파일로 관리하고 앱 시작 시 로드한다. Dynamic Dataset은 SQLite에 저장한다. Event Log는 append-only 방식으로 저장하고, Memory Object와 Review Queue는 Event Log를 기반으로 업데이트된 상태 테이블로 관리한다.

이 방식은 세 가지 장점이 있다. 첫째, Static Dataset을 Git으로 쉽게 관리할 수 있다. 둘째, 학습 이력은 SQLite로 안정적으로 보존할 수 있다. 셋째, 알고리즘 변경 시 Event Log를 재처리하여 상태를 다시 계산할 수 있다.

## 16.5 AI 호출 구조

AI 호출은 UI에서 직접 수행하지 않는다. UI는 AI Tutor Service에 요청하고, AI Tutor Service는 Prompt Library와 Learner Model을 참조하여 프롬프트를 구성한다. 프롬프트는 코드에 하드코딩하지 않고 별도 Markdown 또는 YAML 파일로 관리한다.

AI 호출에는 최소한 다음 정보가 포함된다. 현재 Learning Object, 사용자의 최근 학습 상태, 오답 유형, 원하는 tutor mode, 출력 형식, 금지 행동이다. AI 응답은 가능하면 JSON 구조로 받아 후처리 가능하게 한다.

# Chapter 17. MVP Requirements

## 17.1 MVP 목표

MVP의 목표는 완성형 학습 앱이 아니라 “기억 중심 학습 루프”가 실제로 작동하는지 검증하는 것이다. MVP는 사용자가 앱을 열고, Today Mission을 확인하고, 복습과 회상 중심 문제를 풀고, 학습 이벤트가 기록되며, 다음 Review Queue가 갱신되는 End-to-End 루프를 제공해야 한다.

## 17.2 MVP 필수 기능

| ID | 기능 | 설명 | Acceptance Criteria |
|---|---|---|---|
| MVP-01 | Static Dataset Loader | Learning Object JSON 로드 | 앱에서 전체 Learning Object 목록을 조회할 수 있음 |
| MVP-02 | Today Mission | 오늘 학습 항목 추천 | Review Queue 기반 추천과 이유 표시 |
| MVP-03 | Review Session | 복습 문제 실행 | 최소 3개 문제 유형 지원 |
| MVP-04 | Event Logging | 학습 이벤트 저장 | 모든 답변, 힌트, 스킵, 완료 이벤트 저장 |
| MVP-05 | Mastery Update | 학습 결과 반영 | 정답/오답/응답시간 기반 점수 업데이트 |
| MVP-06 | Review Queue | 다음 복습 예약 | next_review_at 생성 및 정렬 |
| MVP-07 | Weak Object View | 약한 표현 표시 | 낮은 Memory Strength 항목 조회 |
| MVP-08 | AI Tutor Basic | 힌트와 비교 설명 | 정답 즉시 공개 없이 힌트 제공 |
| MVP-09 | Progress Summary | 세션 결과 요약 | 강화/약화/다음 복습 항목 표시 |

## 17.3 MVP 문제 유형

MVP는 최소한 다음 문제 유형을 지원해야 한다. 첫째, meaning recognition은 표현을 보고 의미를 선택하는 문제이다. 둘째, fill blank는 문장 빈칸에 표현을 입력하거나 선택하는 문제이다. 셋째, reverse recall은 한국어 의미 또는 상황을 보고 영어 표현을 직접 입력하는 문제이다. 넷째, contrast choice는 유사 표현 중 문맥에 맞는 표현을 선택하는 문제이다.

초기 MVP에서는 모든 문제를 AI가 실시간 생성할 필요는 없다. Dataset의 Quiz Template을 기반으로 생성하고, AI는 힌트와 설명에 우선 사용한다.

## 17.4 MVP 비기능 요구사항

MVP는 로컬에서 실행 가능해야 한다. 데이터는 쉽게 백업 가능해야 하며, 모든 문서와 스키마는 Git으로 관리되어야 한다. 앱은 복잡한 로그인 없이 단일 사용자 모드로 시작할 수 있어야 한다. 학습 이벤트는 누락 없이 저장되어야 하며, 오류 발생 시 최소한 사용자의 학습 이력이 손실되지 않아야 한다.

# Chapter 18. Roadmap

## 18.1 Phase 0. Documentation Foundation

Phase 0의 목표는 문서 기반 개발 체계를 만드는 것이다. Product Constitution, Product Vision, PRD, Learning Engine Spec, Dataset v2 Spec, Data Model, Implementation Guide를 작성한다. 이 단계의 완료 기준은 Codex가 문서를 읽고 저장소 구조와 기본 구현 계획을 생성할 수 있는 상태이다.

## 18.2 Phase 1. Dataset v2 Foundation

Phase 1에서는 OCR 기반 데이터를 Learning Object 중심으로 재구성한다. 기존 Unit 기반 Markdown과 JSON을 Source Layer로 유지하되, 실제 앱이 사용할 Static Dataset을 별도로 생성한다. 이 단계의 핵심 산출물은 `learning_objects.json`, `source_references.json`, `knowledge_graph_edges.json`, `quiz_templates.json`, `answer_graph.json`이다.

## 18.3 Phase 2. MVP Learning Loop

Phase 2에서는 Today Mission, Review Session, Event Log, Mastery Update, Review Queue를 구현한다. 이 단계가 끝나면 사용자는 실제로 앱을 열고 매일 복습 중심 학습을 수행할 수 있어야 한다.

## 18.4 Phase 3. AI Tutor Integration

Phase 3에서는 AI Tutor를 연결한다. AI Tutor는 힌트 제공, 오답 분석, 유사 표현 비교, 짧은 문장 생성 피드백을 제공한다. AI 응답은 Prompt Library를 통해 통제하고, 학습 이벤트와 연결한다.

## 18.5 Phase 4. Conversation Expansion

Phase 4에서는 표현을 실제 대화 맥락에서 사용하는 기능을 강화한다. 사용자는 특정 표현을 포함한 문장을 만들고, AI는 자연스러움과 문맥 적합성을 평가한다. 이 단계에서 Conversation Success Rate를 Learner Model에 반영한다.

# Chapter 19. Risks & Open Questions

## 19.1 OCR 품질 리스크

원천 PDF는 스캔 기반이므로 OCR 오류가 발생할 수 있다. 특히 phrasal verb, 예문, 문제 번호, 정답지 연결에서 오류가 발생하면 Dataset 품질이 저하된다. 이를 해결하기 위해 OCR Raw와 Source Reference를 보존하고, confidence가 낮은 항목은 수동 검수 대상으로 분리한다.

## 19.2 Dataset 저작권 리스크

사용자는 구매한 교재를 개인 학습 목적으로 활용한다. 그러나 제품 문서와 Dataset은 외부 배포를 전제로 하지 않아야 한다. 저장소는 개인 사용 목적으로 관리하고, 원문 전체 재배포나 공개 배포를 피한다. 앱 내부 데이터는 학습 목적의 구조화 데이터로 관리하되, 외부 공유는 제한한다.

## 19.3 AI 과개입 리스크

AI가 학습자의 외부 개인 정보를 모르는 상태에서 지나치게 적극적으로 개입하면 추천의 신뢰도가 낮아질 수 있다. 따라서 AI 개입은 앱 내부 학습 데이터에 근거해야 하며, 추천 사유를 설명할 수 있어야 한다.

## 19.4 복잡도 증가 리스크

Personal Learning OS라는 비전은 크지만, 초기 구현이 과도하게 복잡해질 위험이 있다. MVP는 반드시 작은 학습 루프에 집중해야 한다. Dataset, Review Queue, Event Log, Mastery Update가 먼저 작동해야 하며, 고급 AI 대화와 음성 기능은 후순위로 둔다.

## 19.5 Open Questions

현재 남아 있는 주요 질문은 다음과 같다. 첫째, 초기 기술 스택을 Next.js 단일 앱으로 갈지 FastAPI 백엔드를 분리할지 결정해야 한다. 둘째, Mastery Score 초기 계산식을 얼마나 단순하게 시작할지 결정해야 한다. 셋째, OCR Dataset v2 변환을 자동화할지 수동 검수 병행으로 할지 결정해야 한다. 넷째, AI Tutor의 첫 구현 범위를 힌트와 설명으로 제한할지 문장 평가까지 포함할지 결정해야 한다.

# Chapter 20. Appendix

## 20.1 주요 용어

| 용어 | 정의 |
|---|---|
| Learning Object | 학습 대상이 되는 최소 정적 지식 단위 |
| Memory Object | 특정 Learning Object에 대한 사용자별 기억 상태 |
| Static Dataset | 사용자 상태와 무관한 고정 지식 데이터 |
| Dynamic Dataset | 학습 이벤트에 따라 변하는 사용자 상태 데이터 |
| Learner Model | 사용자의 학습 및 기억 상태 모델 |
| Adaptive Engine | 다음 학습 행동을 결정하는 추천 엔진 |
| Review Queue | 복습이 필요한 Memory Object 목록 |
| Mastery Score | 표현별 종합 숙련도 |
| Memory Strength | 현재 기억 강도 |
| Forgetting Risk | 가까운 시점에 잊을 가능성 |
| Answer Graph | 문제와 정답을 연결하는 참조 구조 |
